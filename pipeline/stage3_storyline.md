# STAGE 3 — 종합 분석 + 스토리라인 설계 (Storyline Design)

목표: STAGE 1(공개자료)과 STAGE 2(회의록, 있으면)를 통합해 **제안서의 뼈대(스토리라인)**를 확정한다. 이 문서가 STAGE 4 조립의 사양서가 된다.

> AI 기여의 두 번째 판단 레이어. 단순 요약이 아니라 "이 고객을 무엇으로 설득할지" 한 줄로 꿰는 스토리라인을 만든다.

산출물: `output/<고객사>/03_종합_분석.md`

## 입력
- `output/<고객사>/01_공개자료_리서치.md`
- `output/<고객사>/02_회의록_분석.md` (있으면)

## 통합 절차
1. **사실 / 추정 재분리** — 두 문서를 합치며 `[사실]`/`[추정]`을 다시 표시한다. 충돌하면 최신·1차 자료를 우선한다.
2. **문제 구조도** — 페인포인트를 인과 사슬로 잇는다. (예: 반품 증가 → 검수 지연 → 환불 지연 → CS 부하 → 재구매율 압박) 단일 공정 문제가 아니라 연결된 구조임을 보인다.
3. **공개자료 vs 회의록 긴장 지점** — 표면 메시지와 실제 운영 부담의 갭을 짚고, 제안서 반영 방식을 정한다.
4. **의사결정자 맵** — 부서별 KPI를 리터니즈 운영지표와 연결한다. **운영 지표 → KPI 언어 번역**이 핵심(예: 검수 TAT → 환불 리드타임 → CS 재문의율 → 재구매 방어).
5. **제안 기회 우선순위** — 1순위 파일럿을 못 박는다. 중대형 고객은 "좁은 범위 · 기존 3PL 비충돌 · 수치 증명형"이 유리하다.

## 산출 문서 구조
```
# 고객사 종합 분석: <회사명>

## Executive Summary
- 고객사 핵심 상황 / 가장 중요한 니즈 / 가장 큰 페인포인트 / 제안해야 할 방향 / 가장 먼저 제안할 파일럿

## 고객사 구조
| 항목 | 내용 | 제안 관련성 |

## 확인된 니즈와 추정 페인포인트
### 확인된 니즈   | 니즈 | 근거 | 관련 부서 | 제안 연결 |
### 추정 페인포인트 | 페인포인트 | 근거 | 확실성 | 리터니즈 해결 가능성 |

## 문제 구조도와 충돌 지점
<인과 사슬 + 공개자료/회의록 긴장 지점 표>

## 의사결정자 맵
| 부서 | 관심 KPI | 제안 메시지 | 예상 반론 | 대응 |

## 제안 기회 우선순위
| 우선순위 | 제안 | 고객 가치 | 실행 난이도 | 파일럿 적합성 |
### 제안서에 바로 넣을 문장 (3개 내외)

## 추가 확인 필요사항
| 질문 | 왜 필요한가 | 답변에 따라 달라지는 제안 |
```

## 스토리라인 사양 (STAGE 4가 그대로 읽는다)
문서 끝에 아래 블록을 **반드시** 추가한다. STAGE 4는 이 값으로 제안서 placeholder를 채운다.

```
## ▶ STORYLINE SPEC (for STAGE 4)
- PROPOSAL_TITLE: <예: "안다르 B2C 반품 운영 모델 제안">
- COMPANY: <회사명>
- COMPANY_EN: <로고용 영문, 없으면 회사명>
- PLATFORM: <커머스 플랫폼, 예: CAFE24 / 자체몰 / 스마트스토어>
- HERO_HEADLINE: <"왜 지금 X인가" 한 줄>
- HERO_LEAD: <표지 다음 핵심 메시지 2~3문장>
- CARD_고객구조: {title, 3 bullets, pill}
- CARD_핵심과제: {title, 3 bullets, pill}
- CARD_제안방향: {title, 3 bullets, pill}
- STYLE_SKU_NOTE: <SKU 단위 우려 반영 권장안 한 줄>
- PLATFORM_INTEGRATION_NOTE: <단계형 연동 메시지>
- KPI_ROWS: [부서, 관심 KPI, 리터니즈 메시지] × 5~7
- PILOT_PHASES: [기준정렬 / 파일럿운영 / 검증확장] 각 기간·내용
- CHECKLIST: 확인 요청 6항목
- SRC_FOOT: <근거 출처 한 줄>
```
